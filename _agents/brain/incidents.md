# Incident log — what broke, why, and the rule that prevents recurrence

> Pattern memory. Each entry: what happened, the root cause, and the standing rule/check that now guards against it. The advisor reads this to avoid re-recommending things that already bit us, and to check whether the guard is still in place.

## 2026-06-22 — Admin video upload "Server error: 413" (Vercel 4.5 MB function-body limit)
- **What:** Dragging an MP4 into the Live Books / Crucible admin editor's R2 uploader (`BookWorkspace`) failed in production with a bare **`Server error: 413`**. It worked every time on the local dev server, so it read as machine-specific — but it was a deployed-only platform limit.
- **Root cause:** Vercel serverless functions have a **hard ~4.5 MB request-body limit** that app code cannot raise. The old flow POSTed the whole file as `multipart/form-data` to `/api/v2/books/assets/upload`, which then streamed it to R2. Any video over ~4.5 MB was rejected by Vercel's edge **before the route handler ran** → unbranded 413. `next dev` has no such cap, hence "works locally." The `middlewareClientMaxBodySize: '200mb'` in `apps/admin/next.config.ts` is a **red herring** — it governs Next.js *middleware* streaming, not the platform function-body limit.
- **The fix (2026-06-22):** **presigned direct-to-R2 upload** — the browser PUTs the file straight to R2 and never transits Vercel. (1) `packages/core/r2-storage.ts` → new `getPresignedPutUrl(key, contentType)`. (2) New admin-gated endpoint `apps/admin/app/api/v2/books/assets/presign/route.ts` validates type/size against the same tables as the upload route and returns `{presignedUrl, publicUrl, key}` (tiny JSON, never near 4.5 MB). (3) `BookWorkspace.tsx` `upload()` now does presign → `PUT` to R2. Old `/upload` route left intact for small-file callers. **Infra:** added a `PUT` CORS rule on the `canvas-chemistry-assets` bucket for `https://admin.canvasclasses.in` + `http://localhost:3001` (kept the existing GET rule). CORS is live immediately; **the code needs a redeploy** before prod stops 413-ing.
- **The rule now:** **never stream a large user file through a Next.js route handler on Vercel** — anything that can exceed ~4.5 MB must use a presigned direct-to-storage PUT. This applies to *any* uploader, not just video. Note even a 10 MB image cap exceeds the limit, so the other uploaders (`/api/v2/assets/upload`, blog image upload) carry the same latent bug for files >4.5 MB.
- **Follow-up (2026-06-22, same day):** the first fix patched the **wrong path**. The founder's failing widget was **`packages/ui/VideoDropZone.tsx`** (the Crucible question editor's solution-video uploader + Mock Tests), which POSTs `multipart/form-data` to **`/api/v2/assets/upload`** — *not* the `BookWorkspace` path. So prod still 413'd after PR #30 deployed. Lesson: **trace the actual component behind the error string before fixing** — "Crucible admin dashboard" ≠ the books editor; grepping the literal UI error text (`Server error:`) pointed straight at `VideoDropZone`. Second fix migrated that path to the same presign pattern but **preserving the `Asset` doc + audit** (which `/assets/upload` creates and the "Delete Video" flow needs): new `apps/admin/app/api/v2/assets/{presign,register}/route.ts` (presign → browser PUT → register), with the SHA-256 **checksum computed in the browser** (`crypto.subtle`) so server-side dedup survives. No new CORS (same bucket/origins as PR #30). Trade-off: presign uses the canonical `requireAdmin` gate, not the per-chapter `canEditQuestion` RBAC the old route had — fine while the founder is super-admin; revisit if staff get scoped grants.
- **Follow-up #2 — the REAL blocker was CSP, not CORS (2026-06-22):** after both presign fixes deployed, the browser PUT still failed with `Failed to fetch (…r2.cloudflarestorage.com)` on **both** local and prod. Spent two rounds blaming R2 **CORS** — wrong. A direct R2 test (`getSignedUrl` + a real PUT + an OPTIONS preflight, run from a Node script with the live creds) proved **signing, credentials, and bucket CORS were all 100% correct** (preflight 204 with the right `Access-Control-Allow-Origin`; server-side PUT 200). The decisive clue was the **Network tab: the `presign` request showed 200 but there was NO `PUT` entry at all** — the request was killed before hitting the network. Root cause: the admin app's **Content-Security-Policy `connect-src`** (in `apps/admin/next.config.ts`) did not list the R2 S3 endpoint, so the browser refused the cross-origin PUT (CSP blocks show as `Failed to fetch` with no network row + a "Refused to connect" console line). The old proxy upload never tripped this because it POSTed same-origin (`'self'`). **Fix:** added `https://*.r2.cloudflarestorage.com` to `connect-src`. Needs a dev-server restart locally (next.config changes) and a redeploy for prod.
- **The rule now (amended):** when a browser request to an external host "fails to fetch" with **no Network entry**, suspect **CSP `connect-src` before CORS** — CORS failures still produce a network row (red, with a CORS console message); CSP refusals produce no row. Any new direct-to-R2/external upload from the admin app must add its target host to `connect-src` in `apps/admin/next.config.ts`. Verify infra with a standalone Node script using live creds before touching app code or bucket config — it cleanly separates "R2/CORS broken" from "browser policy blocking."
- **Still open:** audit the remaining proxy-through-Vercel uploaders (blog image upload; the `AudioRecorder` if it ever exceeds 4.5 MB) and migrate them to presign if large files are expected; fold the presign → PUT → register flow into a shared helper now that there are two callers (books + assets). Full detail: GBrain `reference/bugs/2026-06-22-video-upload-413-vercel-body-limit.md`.

## 2026-06-19 — canvas-admin Vercel build OOM-killed by the in-app drawing editors
- **What:** The Structure Editor (Ketcher) and Diagram Editor (Excalidraw), built into the admin app at `/structure-editor` + `/diagram-editor`, were committed to `origin/main` but **never went live** — every `canvas-admin` Vercel deploy was OOM-killed (build container ran out of RAM bundling the two heavy editor libraries). The student app deployed fine. A `webpackMemoryOptimizations` flag was tried (commit `5f11823`) but builds still crawled 37+ min and/or errored on the default machine.
- **Root cause:** Two large client libraries (Ketcher's Indigo/WASM + Excalidraw) bundled into a deployed Next app push the build's peak memory past the default Vercel builder's ceiling. The editors are *authoring* tools used only by the founder — they did not need to ship in a public-facing deployed app at all. Putting them there coupled a founder-only convenience to the admin app's deploy health.
- **The fix (2026-06-19, founder decision):** **moved both editors OUT of the deployed app into a standalone local-only tool** — `tools/canvas-draw-studio/` (Vite + React), deliberately placed **outside `apps/*` and `packages/*`** so it never enters any Vercel build. Runs on the founder's machine via a double-click `Start.command` (zero-dependency Node static server over a committed `dist/`); download-to-disk only. Then **removed the editors from the admin app entirely** (routes, feature folders, `/api/v2/iupac`, landing cards, the 4 heavy deps, and the editor-only `next.config` bits incl. the now-moot `webpackMemoryOptimizations`). `build:admin` returns to clean exit 0, no OOM. Commits `34de3a1` (tool) + `8d4e80b` (removal) on branch `feat/local-draw-studio`. Porting to Vite surfaced 3 Webpack-isms Ketcher relies on (stray `require("raphael")`, Node `global`, dev pre-bundle) — see GBrain `reference/2026-06-19-ketcher-excalidraw-on-vite.md`.
- **The rule now:** **founder-only authoring tools that pull in heavy client libraries (Ketcher, Excalidraw, and similar) do NOT belong in a deployed app** (admin or student) — ship them as a standalone local tool outside the workspaces. If a heavy library must live in a deployed app, treat build-memory as a first-class constraint (measure peak RAM; an Enhanced builder is a cost, not a default). Cross-ref: the editor cockpit rows in `_agents/PROJECTS.md`.
- **Still open:** the OOM PR `chore/remove-interactive-graph` should be closed as superseded (its memory-flag fix is now moot); `feat/local-draw-studio` carries everything and is the branch to merge. `gh` CLI not installed — founder closes the PR + merges manually.

## 2026-06-19 — Crucible tag chaos: 828 dangling tags + 55 orphan chapters (now 0)
- **What:** The gap-auditor watchdog found 828 questions tagged with `tag_id`s that don't exist in the taxonomy, + 55 questions in non-existent chapters. Such questions are invisible to tag-based filtering / adaptive practice.
- **Root cause:** **bulk imports tagged questions with ad-hoc, inconsistent strings instead of canonical taxonomy `tag_id`s** — singular/plural drift, old prefixes, free-text concept names, and meaningless "stripe" tags (`tag_goc_12/13/14/15` applied in rotation). The taxonomy also had genuine gaps (no home for several real topics).
- **The fix (2026-06-18/19):** chapter-by-chapter remap to real tags via the reusable `scripts/fix_chapter_dangling.js` (config + assert-targets-exist + abort-if-unmapped + rollback); **5 new primary tags** added where the taxonomy had real gaps (haloalkanes Polyhalogen; aldehyde Prep + Nomenclature; carboxylic Acidity; GOC Reactions & Mechanisms); 42 homeless PYQs soft-deleted (7 chem chapters missing from taxonomy — see CRUCIBLE_STATE), 2 mis-filed moved. **Result: dangling 828 → 0, gap-auditor severity 0.**
- **The rule now:** **all ingestion MUST use canonical taxonomy `tag_id`s** (real topic/micro ids from `taxonomyData_from_csv.ts`), never ad-hoc strings. The gap-auditor watchdog catches regressions. Full detail: GBrain `reference/2026-06-19-crucible-taxonomy-tag-cleanup.md` + CRUCIBLE_STATE changelog.
- **Resolved 2026-06-19 (founder decision):** the 7 "missing" chem chapters (Solid State, Surface Chemistry, Metallurgy, Polymers, Environmental, Everyday Life, s-block) are **NOT being added — they're not in the latest NCERT syllabus**, so the 42 soft-deleted TEMP PYQs **stay permanently deleted (do not restore)**. **Still open:** micro-tagging pass to refine 2 baseline clusters (p-block `tag_pblock_1`, carbonyl image-based Qs).

## 2026-06-10 — Silent hard-deletion of a Live Book section + the founder's video
- **What:** A restructure hard-deleted the entire "Classification of Matter" section on the `introduction-chemistry-matter` page *and unlinked the founder's recorded R2 video*. Unnoticed until 2026-06-14 (4 days).
- **Root cause:** Raw destructive Mongo ops on `book_pages` with no version history, no content-loss guard, and no watchdog. An agent did it; nobody was watching content integrity.
- **The rule now:** CLAUDE.md §0.6 (no hard-delete; mutate only via `scripts/lib/book-writer.js` — version snapshot + content-loss guard + audit + soft-delete; R2 assets never deleted). Content Protection Phases 1 + 1.5 shipped 2026-06-14. The section + original video were restored.
- **Follow-up done (2026-06-18):** the founder advisor's first sweep found **6 pre-§0.6 scripts that still hard-delete `book_pages`** directly (no snapshot/guard) and could erase live content if re-run or copied as templates. All 6 now `require('scripts/lib/legacy-book-guard.js')`, which refuses to run unless `ALLOW_LEGACY_BOOK_SCRIPT=1` is set explicitly and redirects authors to `book-writer.js`. Verified: refuses with exit 1 before any Mongo connection. (The other ~5 grep "hits" were comment-only — verified, not touched.)
- **Still open:** the **daily drift watchdog** (the "content guard" face) — detect a content loss *after the fact* on a schedule. The gateway + the legacy guard are both *preventive*; nothing yet *detects* a loss that slips through. This is what Layer A's content guard closes (being built).

## 2026-06 — Vercel bill spike (~10× outbound traffic)
- **What:** Outbound/Origin-Transfer cost spiked ~10×. Traced to public pages bypassing the edge cache while AI/training bots crawled the sitemap continuously.
- **Root cause:** `force-dynamic` / `revalidate=0` / aggressive `revalidate=60` on public pages; auth reads (`cookies()`) at the layout level flipping pages to dynamic (E132); training crawlers hitting uncached pages (5–8 GB/day, zero return).
- **The rule now:** CLAUDE.md §10 (caching discipline: public pages cacheable, auth in client islands, `revalidate` 3600/86400 defaults) + `robots.ts` blocking training crawlers while allowing answer/citation bots. Closed by commits `6f5af21`, `4e9f188`, `caf8b94` (2026-06-02). "Do not undo."
- **Still open:** the **cost/perf sentinel** (a scheduled check that flags any new `force-dynamic` public page + watches usage) is not yet built. Until then, the founder advisor grep-checks for the forbidden patterns on demand.

## The pattern across both
Same shape: **work happened, the system drifted, and nothing was continuously watching the joint** — so the failure was invisible until it surfaced expensively (a bill) or by luck (a founder noticing missing content). This is the entire case for the AI-native supervisory layer. Each watchdog above is named after the incident it would have caught on day one.
