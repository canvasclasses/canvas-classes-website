# Incident log — what broke, why, and the rule that prevents recurrence

> Pattern memory. Each entry: what happened, the root cause, and the standing rule/check that now guards against it. The advisor reads this to avoid re-recommending things that already bit us, and to check whether the guard is still in place.

## 2026-07-10 — Two Vercel build failures on the same PR (`@types/react` icon-never trap + `AssetType` merge collision)
- **What:** Pushing a large in-flight-work branch (`chore/in-flight-work-2026-07`) to a PR produced two *sequential* red Vercel builds. **(A) STUDENT build:** `Failed to compile … Type error: Type 'number' is not assignable to type 'never'` at `NeetChapterClient.tsx:223` (`<Icon size={18} />`), then again at `organic-chemistry-hub/page.tsx:70`. **(B) ADMIN build (after A was fixed):** `Property 'model' is missing in type '{ image; svg; audio; video; }' but required in type 'Record<AssetType, number>'` at `apps/admin/app/api/v2/books/assets/presign/route.ts:21`.
- **Root cause (A):** `@types/react` **19.2.14** (pinned in `package-lock.json`) tightened `React.ElementType` so a component typed that loosely **rejects numeric props** like `size`/`strokeWidth` — the intersection of the union's prop types collapses to `never`. Any `icon: React.ElementType` field rendered as `<Icon size={n} />` fails. **This is a LATENT issue that lives on `main` too** — the live site runs an older deploy made before the `@types/react` bump, so `main` itself would fail a fresh build until patched. `npx tsc --noEmit` flagged ~10-12 of these, but **tsc is stricter than `next build`** (next build tolerated some, e.g. NewHero's `item.Icon`, that tsc flagged) — so "tsc is clean" ≠ "next build passes" and vice-versa; only a real `next build` is authoritative for deploy.
- **Root cause (B):** a **merge collision between two independent work streams**. `origin/main` added the presign route with `const MAX_SIZE: Record<AssetType, number>` listing 4 keys; the in-flight **anatomy 3D work added a 5th member `'model'` to `AssetType`** in `packages/core/r2-storage.ts` (glTF uploads). After the merge, TS required every `Record<AssetType,…>` map to have a `model` key; the presign map didn't → admin `next build` failed. (The canonical `books/assets/upload/route.ts` already had `model: 60 MB`; it just wasn't mirrored into the new presign route.)
- **The fix (2026-07-10):** **(A)** retyped the 5 offending icon fields from `React.ElementType` → `React.ComponentType<{ size?: number; className?: string; strokeWidth?: number }>` (files: `NeetChapterClient.tsx`, `organic-chemistry-hub/page.tsx`, `QuickRevisionCard.tsx`, `HandwrittenNotesClient.tsx` ×2, `NewHero.tsx`). Type-only, zero visual/behavioural change. **(B)** added `model: 60 * 1024 * 1024` to the presign route's `MAX_SIZE`, matching the upload route. Both apps then built clean locally (`npm run build` + `npm run build:admin`, exit 0).
- **The rule now:** **(1)** Never type an icon/component prop as `React.ElementType` if you render it with `size`/`strokeWidth`/numeric props — use `LucideIcon` or `React.ComponentType<{ size?; className?; strokeWidth? }>`. **(2)** After any `@types/*` bump in the lockfile, run a real `next build` for BOTH apps before trusting a branch — `tsc` alone is not a faithful proxy for Vercel. **(3)** Adding a member to a union like `AssetType` is a **breaking change to every `Record<AssetType,…>` map** — grep `Record<AssetType` (currently: `r2-storage.ts` ASSET_CONFIG, `books/assets/upload` MAX_SIZE, `books/assets/presign` MAX_SIZE) and update all of them in the same commit. **(4)** When merging a long-lived branch, expect exactly these type-exhaustiveness collisions where one side widened a shared union and the other added a new consumer — build both apps post-merge, don't just resolve text conflicts. Cross-ref the standing content-loss / build-safety habits in CLAUDE.md.

## 2026-06-26 — Intermittent "Failed to update question" in the admin editor (a true grey zone)
- **What:** Editing a question's solution/tags in the admin editor (localhost:3001) intermittently popped "❌ Update failed: Failed to update question" with the edit reverted; retrying the SAME edit then worked. Scary because it was non-deterministic — the founder couldn't tell whether a save had actually applied. (Distinct from the same-day GET-strips-solution finding below, which was about the solution *appearing* to vanish; this is the *write* itself intermittently failing.)
- **Root cause:** `packages/services/questions-by-id.ts` PATCH had a fragile shape: **(1)** after the real write (`findByIdAndUpdate`) succeeded, it ran **non-essential post-write steps that were fatal to the response** — `auditLog.save()` (a second DB write, NOT wrapped; the sibling DELETE handler *did* wrap it — inconsistent), analytics, cache revalidation. A transient blip on any of them returned a 500 *even though the edit had saved* → grey zone. **(2)** The read + write had **no retry** for a transient connection drop (Atlas idle-disconnect, laptop sleep/wake, network blip, Next dev recompile) — `connectToDatabase()` guarantees a good socket at request START, but an op issued after a mid-request drop throws immediately (`bufferCommands:false`). **(3)** The catch returned a **generic "Failed to update question"** and logged the real error only to the server console — invisible to a non-technical founder, so every failure was an unsolvable guess. `trackServer` was investigated and RULED OUT (it fails open / never throws).
- **The fix (2026-06-26):** in the PATCH handler — **(a)** a `retryDbOp()` helper retries the `findOne` + `findByIdAndUpdate` up to 3× on *transient* DB errors only (re-establishing the connection between tries; non-transient errors throw immediately — we never paper over real failures); **(b)** all post-write side effects (audit log, analytics, revalidate) moved into a single try/catch — a successful write can NEVER again be reported as failed; **(c)** the catch now surfaces the real error (`details` + a `transient` flag, with a clear "your edit was NOT saved — try again" message) to the operator on localhost-dev or authenticated-admin only (students never reach this write and never get detail — §8.5 preserved; the admin editor already renders `data.details`). Net: the grey zone is eliminated by construction — either it clearly saved, or it clearly says it didn't. Admin+student tsc clean.
- **The rule now:** **a successful primary write must never be coupled to a fatal non-essential step.** Audit-logging / analytics / cache-revalidation after a write are best-effort — wrap them in try/catch, never `await` them un-guarded between the write and the success response. **Admin write handlers should retry transient DB errors** (use `retryDbOp` or the connection-helper philosophy) and **surface the real error to the operator** rather than a generic string that hides the cause.
- **Still open (systemic — same pattern, NOT yet fixed):** the audit-save-fatal-after-write shape also lives in **`apps/admin/app/api/v2/questions/[id]/reclassify/route.ts:171`** (fires when the founder changes a question's *chapter*), **`packages/services/assets-upload.ts:318`**, and **`packages/services/assets-by-id.ts:96`**. Sweep these with the same hardening (wrap audit non-fatally + transient retry + operator-visible errors) so the fix is systemic, not one-route.

## 2026-06-26 — Crucible persistence/refresh audit: timed tests lost on refresh + browse progress invisible after refresh
- **What:** A deep read-only audit of Crucible's persistence behavior (triggered by "what happens if a student refreshes mid-session?") surfaced one CRITICAL and several integrity gaps. **(1) CRITICAL — a timed Test is destroyed by any refresh/tab-evict:** all test state (`answers`, `marked`, timer) lives only in React `useState` in `TestView.tsx`; nothing is written until Submit, and on refresh `CrucibleWizard` remounts with `activeView` reset + `testQuestions=[]`, so the test isn't even re-rendered. The built `recordTestSession` resume helper is **dead code** (`ChapterPracticePage.tsx:40`, never called). **(2) HIGH — browse "lost progress" illusion:** after refresh the browse list re-rendered every solved question as un-attempted (`qStates` never re-seeded from the progress GET), so students re-answered questions they'd already done. **(3) MEDIUM integrity — client-trusted correctness:** `progress/batch` and `session-response` wrote client-sent `is_correct`/`answeredCorrectly` straight into the persona mastery counters with no server recompute (the `test-results` route was hardened for this in Audit #8; its siblings were not). Plus: no loading skeleton (a misleading "0 answered" flash on every refresh), `all_attempted_ids` uncapped (slow-burn toward the 16 MB BSON ceiling = eventual total write-outage for a power user), no idempotency keys (beacon+keepalive double-fire double-counts), and filters/scroll/position lost on refresh (only chapter+mode+examBoard are in the URL).
- **Root cause:** persistence was designed *per-attempt* (immediate POST + beacon backstop — the May-2026 tab-close lesson, correctly applied in BrowseView) but **session/UI restoration on reload was never built**, and **two of three write paths skipped the server-side correctness recompute** that the test-results route established. The DB was almost always correct; the UI just failed to *reflect* it after a reload, which reads as data loss.
- **The fix (2026-06-26, this session — partial, safe subset):** **(H1)** BrowseView now re-seeds `qStates` from the mount progress GET so solved questions come back locked + marked correct/wrong (merge-only, never clobbers in-session edits). **(M1)** added a `progressLoaded` flag that shows `–` instead of a confident `0` for solved/answered counts until the GET resolves. **(C2)** `progress/batch/route.ts` now recomputes `is_correct` server-side via the shared `isAnswerCorrect` helper (fetch canonical `QuestionV2`, override every attempt) + added the missing 200-item batch cap. **(C1 mitigation)** `TestView` gained a `beforeunload` guard so an accidental refresh during an in-progress test triggers the browser's native "Leave site?" prompt — kills the most common loss trigger without touching the fragile test-generation path. All type-clean + lint-clean. Caveat that started the session: this audit was prompted by a *false alarm* — a LaTeX backslash fix the founder thought broke tag-saving; it didn't (a separate localhost-only GET-strips-solution bug, fixed same day in `questions-by-id.ts`).
- **The rule now:** **(a)** every new practice mode must restore its answered/UI state on reload, not just persist the attempt — server truth that the UI can't reflect reads as data loss. **(b)** *Every* attempt write path must recompute correctness server-side from canonical `QuestionV2` (never trust client `is_correct`) — mirror `test-results`/`progress/batch`, never the old client-trust pattern. **(c)** new per-user array fields on `UserProgress` need a cap from day one. Cross-ref CRUCIBLE_ARCHITECTURE.md §3.2 (persistence contract) + §10 (don't trust client score).
- **Follow-up shipped (2026-06-26, second pass — points 2/3/4 of the still-open list):** **(2) guided server-side recompute** — `session-response/route.ts` now recomputes correctness via `isAnswerCorrect` from canonical `QuestionV2`; the guided client (`AdaptiveQuestionCard` → `AdaptiveSession.writeResponseToAPI`) now sends the raw `selectedOption` (backward-compatible: server falls back to the client value only when the selection wasn't sent or the question is missing). **(3) caps** — `ALL_ATTEMPTED_IDS_CAP` (20,000, LRU by `last_attempted_at`) and `STARRED_QUESTIONS_CAP` (2,000, FIFO) added in `packages/persona/contract.ts`; eviction lives in `computeUserProgressUpdate` (all_attempted_ids) and the starred route. **(4a) idempotency** — added optional `client_attempt_id` to `IQuestionAttempt`; the updater returns `skipped:true` for a duplicate in the `recent_attempts` window and `writer.ts` early-returns; BrowseView (per-submit uuid, reused by the beacon replay) + TestView (`t{testStartTime}_{qid}`, stable across retries) + both routes now stamp/forward it. **(4c) FIFO bookmarks** = the starred cap above. **(4b) browse filter URL-sync** — BrowseView mirrors `f_exam`/`f_year`/`f_topic`/`q` into the query string (merge-preserving the wizard's `mode`/`examBoard`; the wizard's URL writer was made merge-preserving too) and seeds from them on mount, so a refresh keeps the student's filters. All type-clean (0 new tsc errors) + lint-clean. Architecture doc §4.1/§4.1.1 updated. **Browser-verification still pending** (founder runs their own dev server).
- **Still open:** **full timed-test resume** (point 1 — being handled in a separate chat; spans the high-blast-radius `CrucibleWizard` generation flow). Lower-priority remainders: browse **scroll-position / current-question** restore (only filters were synced — scroll restore on a windowed list is a bigger lift); **server-UTC streak day-bucketing** (off-by-one for late-evening IST — `user-progress-updater.ts:229`); admin `getUserProgressSummary` reads dead field `total_attempted` on `concept_mastery` (real field `times_attempted`) so admin weak-areas is silently always-empty; and **10 pre-existing `never`-type build errors** in unrelated files (NewHero, QuickRevisionCard, HandwrittenNotesClient, NeetChapterClient, organic-chemistry-hub) — confirmed present on committed code independent of this work, will break `npm run build`. Note: the architecture-doc "streak broken (audit #18)" claim was already corrected to "fixed" in the first pass.
## 2026-06-22 — Admin video upload "Server error: 413" (Vercel 4.5 MB function-body limit)
- **What:** Dragging an MP4 into the Live Books / Crucible admin editor's R2 uploader (`BookWorkspace`) failed in production with a bare **`Server error: 413`**. It worked every time on the local dev server, so it read as machine-specific — but it was a deployed-only platform limit.
- **Root cause:** Vercel serverless functions have a **hard ~4.5 MB request-body limit** that app code cannot raise. The old flow POSTed the whole file as `multipart/form-data` to `/api/v2/books/assets/upload`, which then streamed it to R2. Any video over ~4.5 MB was rejected by Vercel's edge **before the route handler ran** → unbranded 413. `next dev` has no such cap, hence "works locally." The `middlewareClientMaxBodySize: '200mb'` in `apps/admin/next.config.ts` is a **red herring** — it governs Next.js *middleware* streaming, not the platform function-body limit.
- **The fix (2026-06-22):** **presigned direct-to-R2 upload** — the browser PUTs the file straight to R2 and never transits Vercel. (1) `packages/core/r2-storage.ts` → new `getPresignedPutUrl(key, contentType)`. (2) New admin-gated endpoint `apps/admin/app/api/v2/books/assets/presign/route.ts` validates type/size against the same tables as the upload route and returns `{presignedUrl, publicUrl, key}` (tiny JSON, never near 4.5 MB). (3) `BookWorkspace.tsx` `upload()` now does presign → `PUT` to R2. Old `/upload` route left intact for small-file callers. **Infra:** added a `PUT` CORS rule on the `canvas-chemistry-assets` bucket for `https://admin.canvasclasses.in` + `http://localhost:3001` (kept the existing GET rule). CORS is live immediately; **the code needs a redeploy** before prod stops 413-ing.
- **The rule now:** **never stream a large user file through a Next.js route handler on Vercel** — anything that can exceed ~4.5 MB must use a presigned direct-to-storage PUT. This applies to *any* uploader, not just video. Note even a 10 MB image cap exceeds the limit, so the other uploaders (`/api/v2/assets/upload`, blog image upload) carry the same latent bug for files >4.5 MB.
- **Follow-up (2026-06-22, same day):** the first fix patched the **wrong path**. The founder's failing widget was **`packages/ui/VideoDropZone.tsx`** (the Crucible question editor's solution-video uploader + Mock Tests), which POSTs `multipart/form-data` to **`/api/v2/assets/upload`** — *not* the `BookWorkspace` path. So prod still 413'd after PR #30 deployed. Lesson: **trace the actual component behind the error string before fixing** — "Crucible admin dashboard" ≠ the books editor; grepping the literal UI error text (`Server error:`) pointed straight at `VideoDropZone`. Second fix migrated that path to the same presign pattern but **preserving the `Asset` doc + audit** (which `/assets/upload` creates and the "Delete Video" flow needs): new `apps/admin/app/api/v2/assets/{presign,register}/route.ts` (presign → browser PUT → register), with the SHA-256 **checksum computed in the browser** (`crypto.subtle`) so server-side dedup survives. No new CORS (same bucket/origins as PR #30). Trade-off: presign uses the canonical `requireAdmin` gate, not the per-chapter `canEditQuestion` RBAC the old route had — fine while the founder is super-admin; revisit if staff get scoped grants.
- **Follow-up #2 — the REAL blocker was CSP, not CORS (2026-06-22):** after both presign fixes deployed, the browser PUT still failed with `Failed to fetch (…r2.cloudflarestorage.com)` on **both** local and prod. Spent two rounds blaming R2 **CORS** — wrong. A direct R2 test (`getSignedUrl` + a real PUT + an OPTIONS preflight, run from a Node script with the live creds) proved **signing, credentials, and bucket CORS were all 100% correct** (preflight 204 with the right `Access-Control-Allow-Origin`; server-side PUT 200). The decisive clue was the **Network tab: the `presign` request showed 200 but there was NO `PUT` entry at all** — the request was killed before hitting the network. Root cause: the admin app's **Content-Security-Policy `connect-src`** (in `apps/admin/next.config.ts`) did not list the R2 S3 endpoint, so the browser refused the cross-origin PUT (CSP blocks show as `Failed to fetch` with no network row + a "Refused to connect" console line). The old proxy upload never tripped this because it POSTed same-origin (`'self'`). **Fix:** added `https://*.r2.cloudflarestorage.com` to `connect-src`. Needs a dev-server restart locally (next.config changes) and a redeploy for prod.
- **The rule now (amended):** when a browser request to an external host "fails to fetch" with **no Network entry**, suspect **CSP `connect-src` before CORS** — CORS failures still produce a network row (red, with a CORS console message); CSP refusals produce no row. Any new direct-to-R2/external upload from the admin app must add its target host to `connect-src` in `apps/admin/next.config.ts`. Verify infra with a standalone Node script using live creds before touching app code or bucket config — it cleanly separates "R2/CORS broken" from "browser policy blocking."
- **Still open:** audit the remaining proxy-through-Vercel uploaders (blog image upload; the `AudioRecorder` if it ever exceeds 4.5 MB) and migrate them to presign if large files are expected; fold the presign → PUT → register flow into a shared helper now that there are two callers (books + assets). Full detail: GBrain `reference/bugs/2026-06-22-video-upload-413-vercel-body-limit.md`.

If a heavy library must live in a deployed app, treat build-memory as a first-class constraint (measure peak RAM; an Enhanced builder is a cost, not a default). Cross-ref: the editor cockpit rows in `_agents/PROJECTS.md`.
- **Still open:** the OOM PR `chore/remove-interactive-graph` should be closed as superseded (its memory-flag fix is now moot); `feat/local-draw-studio` carries everything and is the branch to merge. `gh` CLI not installed — founder closes the PR + merges manually.

## The pattern across both
Same shape: **work happened, the system drifted, and nothing was continuously watching the joint** — so the failure was invisible until it surfaced expensively (a bill) or by luck (a founder noticing missing content). This is the entire case for the AI-native supervisory layer. Each watchdog above is named after the incident it would have caught on day one.
